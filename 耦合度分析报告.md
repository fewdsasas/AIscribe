# AIscribe 项目前后端耦合度分析报告

> 分析日期：2026-06-26
> 分析对象：Electron 三进程（main / preload / renderer）的代码、类型与 IPC 依赖

---

## 一、整体评估

| 维度 | 评分（1-10，越高越耦合） | 说明 |
|------|------------------------|------|
| **整体架构耦合度** | **5.5 / 10** | 中等耦合：IPC 集中、类型共享合理，但 renderer 对 `window.aiscribe` 的 API 面存在大面积直接依赖 |
| 目录/编译隔离 | 2 / 10 | 三个进程入口与路径别名清晰，无跨进程直接 import |
| IPC 接口耦合 | 7 / 10 | 35+ channel、40+ API 方法，renderer 18 个组件/视图直接调用 |
| 数据类型共享 | 4 / 10 | 核心领域类型通过 `@shared` 共享，DTO 集中在 `ipc.ts`，但存在少量 renderer 不需要的类型 |
| 状态管理耦合 | 4 / 10 | renderer store 仅持有 UI 状态，不直接持有 main 状态 |
| 单例/生命周期耦合 | 6 / 10 | main 中 Database/SkillLoader/LearningEngine 为单例，renderer 只能通过 IPC 间接依赖 |

**结论**：项目整体耦合度处于**中等偏可控**水平。主要的可优化点集中在 **IPC API 面的膨胀** 和 **renderer 组件与 main 能力的紧耦合**。

## 二、进程依赖矩阵

```
                 main          preload        renderer
main             ──            注册 handlers   无直接依赖
preload          通过 IPC 调用  ──             expose window.aiscribe
renderer         无直接依赖     通过 window.aiscribe  ──
shared/types     main 大量引用  preload 全部引用  renderer 大量引用
```

| 依赖方向 | 是否允许 | 现状 | 评价 |
|---------|---------|------|------|
| renderer → main | ❌ 不应直接 | 无直接 import ✅ | 优秀 |
| main → renderer | ❌ 不应直接 | 无直接 import ✅ | 优秀 |
| renderer → preload | 通过 `window.aiscribe` | 18 个组件/视图直接调用 | 需要封装层 |
| main/shared | 通过 `@shared` | 27 个文件引用 shared types | 合理 |
| preload/shared | 通过 `@shared` | preload 仅引用 shared | 优秀 |

## 五、解耦建议

### 5.1 高优先级

| 编号 | 建议 | 收益 | 涉及文件 |
|------|------|------|---------|
| H1-1 | 在 renderer 中创建 `services/llmStreamService.ts`，封装 start/cancel/onChunk/onDone/onError/remove | 将流式协议从 UI 组件中移除，AIChatView 只接收 `streamingText`/`error` | AIChatView.tsx |
| H2-1 | 删除 `chatSlice.ts` 中的 `getAiscribe()` cast，使用 typed `window.aiscribe`；或注入 api 实例 | 类型安全，方便 mock | chatSlice.ts |
| H3-1 | 在 renderer 创建 `services/projectService.ts`，App.tsx 通过 hook 使用 | 降低顶层组件与 IPC 的耦合 | App.tsx |
| H4-1 | 将 `getDb/getSkillLoader/getLearningEngine` 抽象为 `ServiceRegistry`，支持依赖注入 | 提升 main 模块可测试性 | main/ipc/index.ts |
| H5-1 | 将 LLM 监听器改为按订阅 ID 管理（而非全局单例），支持多组件同时订阅 | 避免组件间互相覆盖 | preload/index.ts |

### 5.2 中优先级

| 编号 | 建议 | 收益 | 涉及文件 |
|------|------|------|---------|
| M1-1 | 新增 IPC `search:global`，在 main 中聚合搜索 | 减少 N+1 IPC，降低网络/序列化开销 | GlobalSearch.tsx, main |
| M2-1 | `projectDashboardStats` 返回已聚合的 totalChapters | 移除 UI 层的 reduce 逻辑 | DashboardView.tsx, project.ipc.ts |
| M3-1 | 新增 `editor:loadNovel(novelId)` 或 `editor:loadByProject(projectId)` 聚合接口 | 简化 EditorView 初始化 | EditorView.tsx, novel.ipc.ts |
| M4-1 | 将 `ipc.ts` 拆分为 `ipc/channels.ts`、`ipc/dto.ts`、`ipc/api.ts` | 单一职责，降低维护成本 | shared/types/ipc.ts |
| M5-1 | 将 renderer 未使用的类型（TrajectoryEntry、SessionMemory 等）迁移到 `main/types.ts` | 减少 shared 包体积与编译依赖 | shared/types/index.ts |

### 5.3 低优先级 / 低风险重构

- 为 renderer 创建 `services/` 目录统一封装所有 `window.aiscribe` 调用
- 引入 React Query / SWR 风格的 `useIPCQuery` hook，统一 loading/error/retry
- 将 `electron.d.ts` 中的 `AiscribeAPI` 按 domain 拆分为 `ProjectAPI`、`ChapterAPI`、`LLMAPI` 等

## 七、结论

AIscribe 项目的前后端隔离**基本良好**：

1. ✅ **无跨进程直接 import**：renderer 不引用 `@main/*`，main 不引用 `@renderer/*`
2. ✅ **IPC 集中管理**：channel 常量和 DTO 都集中在 `shared/types/ipc.ts`
3. ✅ **preload 薄桥接**：只负责暴露 API，无业务逻辑
4. ⚠️ **renderer 对 IPC API 存在大面积直接依赖**：18 个组件/视图、48 处调用
5. ⚠️ **部分业务逻辑泄漏到 UI 层**：如 GlobalSearch 的 N+1 调用、Dashboard 的 reduce
6. ⚠️ **main 单例生命周期与 IPC 注册耦合**：`getDb/getSkillLoader/getLearningEngine`

**推荐下一步行动**：

1. 创建 `src/renderer/services/` 封装 `window.aiscribe`，按 domain 拆分（project、chapter、llm、search）
2. 新增聚合型 IPC 接口（`search:global`、`editor:loadNovel`）减少 UI 层 N+1 调用
3. 将 `shared/types/ipc.ts` 拆分为 channel/dto/api 三个文件
4. 将 renderer 未使用的内部类型迁回 main 层

预计重构后整体耦合度可从 **5.5/10** 降至 **3.5/10**。
