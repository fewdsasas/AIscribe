# AIscribe 模块化解耦实施计划

## 一、Summary

本计划采用**渐进式解耦**策略，在不破坏现有功能的前提下，将 AIscribe 项目从当前紧耦合的三进程架构逐步改造为：

- **数据层（Data Layer）**：Repository + Database，通过接口抽象
- **业务逻辑层（Business Layer）**：LLM Engine、Learning Engine、Skill Loader 等依赖接口而非具体实现
- **接口层（API/IPC Layer）**：Handler 通过依赖注入获取服务，不直接实例化业务对象
- **表现层（Presentation Layer）**：Renderer 通过 service/hook 封装访问 `window.aiscribe`，不直接调用 IPC API

目标：整体耦合度从 **5.5/10 降至 3.5/10**，提升可测试性、可维护性和可扩展性。

## 二、Current State Analysis

### 2.1 项目分层现状

| 层级 | 目录 | 现状 | 问题 |

## 三、Proposed Changes

### Phase 1：Main 进程依赖注入基础设施（基础层）

**目标**：建立 `ServiceRegistry`，让 IPC handler 通过注册表获取服务，而非直接访问全局单例。

#### 3.1.1 创建服务接口定义

**文件**：`src/main/di/service-interfaces.ts`（新增）

- 定义 `IDatabase` 接口，提取 `Database` 中 IPC handler 需要的公开方法（save、close、各仓库访问器）
- 定义 `ISkillLoader` 接口，提取 `SkillLoader` 的 `list()`、`get(name)`、`invoke(name, input)`
- 定义 `ILearningEngine` 接口，提取 `recordInteraction`、`analyzeProject`、`getProjectSummary`、`close`
- 定义 `ILLMProvider` 接口，提取 `chat`、`chatStream`、`cancelStream`、`resetConfig`

**原因**：让业务逻辑层依赖抽象接口，便于 mock 和替换实现。

#### 3.1.2 创建 ServiceRegistry

**文件**：`src/main/di/service-registry.ts`（新增）

- 提供 `register<T>(token: string, factory: () => T)` 和 `resolve<T>(token: string): T`
- 内置 token 常量：`DATABASE_TOKEN`、`SKILL_LOADER_TOKEN`、`LEARNING_ENGINE_TOKEN`、`LLM_PROVIDER_TOKEN`
- 提供 `createDefaultServices()` 工厂函数，按现有方式初始化 Database/SkillLoader/LearningEngine/LLMProvider
- 提供 `setTestServices()` 用于测试注入 mock

**原因**：集中管理依赖生命周期，替代模块级单例。

#### 3.1.3 改造 `src/main/ipc/index.ts`

- 删除模块级单例 `db`、`dbInitPromise`、`skillLoader`、`learningEngine`、`learningInitPromise`
- 删除 `getDb()`、`getSkillLoader()`、`getLearningEngine()` 函数
- `registerIpcHandlers` 接收可选的 `services?: Partial<ServiceRegistry>` 参数，默认调用 `createDefaultServices()`
- handler 注册时通过 `services.resolve(DATABASE_TOKEN)` 等获取服务

**涉及文件**：
- `src/main/ipc/index.ts`
- `src/main/ipc/chat.ipc.ts`（改为接收 registry 或服务参数）
- `src/main/ipc/skill.ipc.ts`
- `src/main/ipc/learning.ipc.ts`
- `src/main/ipc/db.ipc.ts`
- `src/main/ipc/project.ipc.ts` 等（视需要逐步迁移）

#### 3.1.4 改造 `src/main/index.ts`

- `registerIpcHandlers(ipcMain)` 保持不变（内部使用默认 registry）
- `LLMProvider.initFromStorage()` 改为通过 registry 初始化

---

### Phase 2：数据层接口化

**目标**：Database 和各 Repository 通过接口交互，降低 `BaseRepository` 与 `Database` 的耦合。

#### 3.2.1 创建数据层接口

**文件**：`src/main/memory/repositories/repository-interfaces.ts`（新增）

- 定义 `IChapterRepository`、`IProjectRepository`、`INovelRepository` 等接口
- 每个接口包含当前 `Repository` 的公开方法签名

#### 3.2.2 Database 实现 `IDatabase` 接口

- `Database` 类实现 `IDatabase`（在 Phase 1 中定义）
- 各 Repository 构造函数接收 `IDatabase` 或底层 `SqlJsDatabase` + `OperationLog`，而非直接 import `Database` 类

#### 3.2.3 Repository 实现接口

- `ChapterRepository implements IChapterRepository`
- `ProjectRepository implements IProjectRepository`
- 其他 Repository 类似处理

**原因**：让业务层依赖 `IChapterRepository` 而非 `ChapterRepository` 具体类。

---

### Phase 3：业务逻辑层解耦

#### 3.3.1 LLMProvider 去静态单例

**文件**：`src/main/engine/llm-provider.ts`

- 将静态方法改为实例方法
- 保留 `static defaultInstance` 作为兼容入口，但新增 `LLMProviderFactory`
- `ILLMProvider` 接口在 ServiceRegistry 中注册
- `activeControllers` 改为实例字段

#### 3.3.2 LearningEngine 依赖接口

**文件**：`src/main/learning/engine.ts`

- 构造函数接收 `IDatabase` 而非 `Database`
- `TrajectoryRecorder` 依赖 `ITrajectoryRepository` 接口

#### 3.3.3 SkillLoader 依赖接口

**文件**：`src/main/engine/skill-loader.ts`

- 通过 `ISkillLoader` 接口暴露
- 内部文件系统操作保持不变

---

### Phase 4：IPC Handler 依赖注入改造

**目标**：每个 handler 文件接收服务接口参数，不直接 import Database/Engine。

#### 3.4.1 逐个迁移 handler 文件

按优先级顺序：

1. `src/main/ipc/learning.ipc.ts`（依赖 LearningEngine）
2. `src/main/ipc/skill.ipc.ts`（依赖 SkillLoader）
3. `src/main/ipc/chat.ipc.ts`（依赖 LLMProvider）
4. `src/main/ipc/db.ipc.ts`（依赖 Database）
5. `src/main/ipc/project.ipc.ts`（依赖 Database）
6. 其他 handler 文件

**改造方式**：
- 原 `registerXxxHandlers(ipcMain: IpcMain)` 改为 `registerXxxHandlers(ipcMain: IpcMain, services: ServiceRegistry)`
- handler 内部通过 `services.resolve(DATABASE_TOKEN)` 获取服务
- 保留 wrap/wrapEvent 不变

---

### Phase 5：Renderer 层服务封装

**目标**：Renderer 组件不直接调用 `window.aiscribe`，通过 service 和 hook 封装。

#### 3.5.1 创建 `src/renderer/services/` 目录

新增文件：
- `src/renderer/services/projectService.ts`：封装 projectList、projectGet、projectCreate、projectUpdate、projectDelete
- `src/renderer/services/chapterService.ts`：封装 chapterList、chapterGet、chapterUpdate、chapterCreate
- `src/renderer/services/novelService.ts`：封装 novelGet、novelGetByProject、novelCreate
- `src/renderer/services/llmService.ts`：封装 startLLMStream、cancelLLMStream、onLLMChunk/onLLMDone/onLLMError/removeLLMListeners
- `src/renderer/services/searchService.ts`：封装全局搜索逻辑（先使用现有 IPC，未来可切换到 `search:global`）

#### 3.5.2 创建通用 hook

**文件**：`src/renderer/hooks/useIPC.ts`（新增）

- `useIPCQuery<T>(fn: () => Promise<T>, deps: unknown[]): { data, loading, error, refetch }`
- `useIPCMutation<T, A>(fn: (args: A) => Promise<T>): { mutate, loading, error }`

#### 3.5.3 替换组件中的直接调用

按优先级顺序：

1. `src/renderer/App.tsx`：使用 `projectService.getProject`
2. `src/renderer/views/DashboardView.tsx`：使用 `projectService`
3. `src/renderer/views/EditorView.tsx`：使用 `novelService` + `chapterService`
4. `src/renderer/components/ai-chat/AIChatView.tsx`：使用 `llmService`
5. `src/renderer/components/search/GlobalSearch.tsx`：使用 `searchService`
6. 其他组件视情况替换

**注意**：由于改动面大，Phase 5 可以分多轮进行，每轮只替换 1-2 个组件。

### Phase 6：类型与模块边界整理

#### 3.6.1 拆分 `src/shared/types/ipc.ts`

新增文件：
- `src/shared/types/ipc/channels.ts`：仅 IPC_CHANNELS 常量
- `src/shared/types/ipc/dto.ts`：CreateXxxData、UpdateXxxData 等 DTO
- `src/shared/types/ipc/api.ts`：SkillListItem、ExportResult 等 API 返回类型

**修改**：

#### 3.6.2 迁移 renderer 未使用的内部类型

- 将 `TrajectoryEntry` 从 `src/shared/types/index.ts` 迁移到 `src/main/learning/types.ts`
- 将 `SessionMemory` 从 `src/shared/types/index.ts` 迁移到 `src/main/memory/types.ts`（如 renderer 确实未使用）
- 检查其他 renderer 未使用的类型并迁移

**注意**：迁移前需确认没有 renderer 文件引用这些类型。

### Phase 7：访问控制与边界保护

#### 3.7.1 强化 tsconfig 路径别名约束

- 保持现有 `@main/*`、`@renderer/*` 别名
- 可考虑使用 ESLint `no-restricted-imports` 规则禁止：
  - `src/renderer/**` 不能 import `src/main/**`
  - `src/main/**` 不能 import `src/renderer/**`
  - `src/preload/**` 只能 import `src/shared/**`

**文件**：`.eslintrc` 或新增 `.eslintrc.cjs` 的 `rules` 配置

#### 3.7.2 模块可见性约定

- 每个模块目录下创建 `index.ts` 作为唯一公开出口
- 内部实现文件不直接暴露给其他模块
- 通过 `internal/` 子目录标记内部实现

## 四、Assumptions & Decisions

### 4.1 采用渐进式而非激进式

**决策**：分 7 个 Phase 逐步推进，每个 Phase 完成后运行全部测试。

**理由**：
- 项目已有 606 个测试，激进重构风险过高
- 渐进式可以在每个阶段获得反馈
- 便于在资源有限时暂停和恢复

### 4.2 先解耦 Main 进程，再解耦 Renderer

**决策**：Phase 1-4 先完成 main 进程的 DI，Phase 5 再改造 renderer。

**理由**：
- Main 进程是业务核心，解耦收益最大
- Renderer 的 service 封装可以建立在稳定的 IPC 层之上
- Main 进程的测试更容易验证（真实 DB + mock）

### 4.3 保留 `window.aiscribe` 全局 API

**决策**：不解除 preload 暴露的 API，而是在 renderer 内部增加 service 封装层。

**理由**：
- 完全替换 IPC API 成本高、风险大
- 增加 service 层已经能显著降低耦合
- 未来如需替换通信方式（如 REST、WebSocket），只需修改 service 层

### 4.4 ServiceRegistry 采用简单 DI 而非重量级框架

**决策**：手写轻量级 registry，不使用 tsyringe/inversify 等框架。

**理由**：
- 项目规模不需要完整 IoC 容器
- 手写 registry 可控、易理解、无额外依赖
- 满足当前解耦需求

### 4.5 接口命名约定

- 服务接口：`IXxxService` 或 `IXxx`（如 `IDatabase`、`ILLMProvider`）
- Token 常量：`XXX_TOKEN`（如 `DATABASE_TOKEN`）
- Registry 方法：`register<T>(token, factory)`、`resolve<T>(token)`

## 七、Risk & Mitigation

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 重构引入回归 bug | 高 | 每个 Phase 后运行 606+ 测试；小步提交 |
| TypeScript 类型冲突 | 中 | 使用 `satisfies`、保留 barrel export 兼容 |
| 测试需要大量更新 | 中 | mock 工厂统一放在 `tests/mocks/services.ts` |
| Renderer service 层过度抽象 | 低 | 只对高频/复杂调用封装，简单调用可保留 |
| 工期超出预期 | 中 | 可分阶段交付，Phase 4 完成后 main 进程已显著改善 |
