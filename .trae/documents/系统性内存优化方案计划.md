# 系统性内存优化方案 — 实施计划

## 摘要

基于对 AIscribe（Electron + React + sql.js WASM + TipTap）的深度内存架构分析，识别出 8 个模块的 25+ 个内存风险点。本计划从数据结构、算法效率、资源管理、缓存策略、监控体系 5 个维度实施优化，目标降低内存占用 30%+、减少 GC 频率 40%+。

## 当前状态分析

### 内存风险热力图

| 模块 | 风险等级 | 核心问题 |
|------|---------|---------|
| 数据库 (sql.js WASM) | **高** | 整库常驻 WASM 堆无法归还 OS；save() 全量 export 三倍峰值；OperationLog pending 无上限；trajectories 表无限增长 |
| AIChatView | **高** | `useAppStore()` 无 selector 全订阅致流式高频重渲染；渲染端 abort 无法取消 main fetch |
| LLM Provider | **中** | SSE buffer 字符串拼接 O(n²)；done 分支无显式 reader.releaseLock |
| TipTap 编辑器 | **中** | WritingGoal `_debounceTimer` 未在 destroy 清理；simulated streaming 不可取消 |
| React 状态 | **中** | pendingMessages 流式期间过时；对象 selector 无 shallow compare |
| IPC 数据传递 | **中** | 全结构化克隆；chapterList 返回全部 content；getProjectTrajectories(_, 10000) 大查询；导出无流式 |
| 资源管理 | **中** | chatSlice saveTimer 无 unload flush；多处 setTimeout 不可取消 |
| 缓存机制 | **低** | 完全无 DB 查询缓存；无 WeakMap/WeakRef |

## 提议的变更

### 阶段 1：P0 紧急修复（高影响、可并行）

#### 任务 1.1：修复渲染端 abort 无法取消 main 进程 fetch
- **文件**: `src/main/engine/llm-provider.ts`、`src/main/ipc/chat.ipc.ts`、`src/preload/index.ts`、`src/renderer/components/ai-chat/AIChatView.tsx`
- **问题**: 用户点"停止"后 main 进程 fetch 流仍在继续，浪费 token 和带宽
- **方案**:
  1. LLMProvider 增加 `private activeControllers: Map<string, AbortController>`，`chatStream` 接受 `requestId` 参数
  2. 新增 `cancelStream(requestId)` 方法，调用 `controller.abort()`
  3. chat.ipc.ts 新增 `llm:cancel-stream` handler，接受 requestId
  4. preload 暴露 `cancelLLMStream(requestId)` 方法
  5. AIChatView 的 abortRef 改为存储 requestId，停止时调用 `cancelLLMStream(requestId)`
- **验证**: 测试 abort 后 main 进程 fetch 确实被取消（mock fetch 验证 abort signal）

#### 任务 1.2：修复 AIChatView 全订阅 + 对象 selector
- **文件**: `src/renderer/components/ai-chat/AIChatView.tsx`
- **问题**: `useAppStore()` 无 selector 订阅整个 store，流式期间 streamingContent 高频更新导致整个组件重渲染
- **方案**:
  1. 行 100-112 的 `useAppStore()` 拆分为多个独立 selector
  2. 行 95-99 的对象 selector 使用 `useShallow`（zustand 提供的 shallow compare）
  3. 流式期间只订阅 `streamingContent` 和 `isStreaming`
- **验证**: 验证流式期间非相关组件不重渲染

### 阶段 2：数据库内存优化（高影响）

#### 任务 2.1：优化 save() 内存峰值
- **文件**: `src/main/memory/database.ts`
- **问题**: save() 调用 `sqlDb.export()` 产生完整 Uint8Array，再 `Buffer.from()` 复制一份，三倍峰值
- **方案**:
  1. export 后直接写 Buffer，避免二次复制：`const data = this.sqlDb.export(); fs.writeFileSync(tmpPath, Buffer.from(data))` → 改为 `fs.writeFileSync(tmpPath, data as unknown as Buffer)`（sql.js export 返回 Uint8Array，Node.js fs.writeFileSync 接受 ArrayBufferView）
  2. 添加 `PRAGMA temp_store=MEMORY` 减少临时文件 IO
  3. 添加 `PRAGMA soft_heap_limit=50000000`（50MB）限制 sql.js 内存使用
- **验证**: 测试 save() 前后 process.memoryUsage().heapTotal 变化

#### 任务 2.2：OperationLog 改为定长环形缓冲
- **文件**: `src/main/memory/operation-log.ts`
- **问题**: `pending: string[]` 无上限，极端场景下数组无限增长
- **方案**:
  1. 添加 `MAX_PENDING = 1000` 常量
  2. `append()` 中检查 `pending.length >= MAX_PENDING` 时触发同步 flush
  3. replay() 改为逐行读取（`readline` 模块）而非 `readFileSync().split('\n')`
- **验证**: 测试 2000 条操作时 pending 不超过 1000

#### 任务 2.3：trajectories 表 TTL 清理
- **文件**: `src/main/memory/database.ts`（migration）、`src/main/learning/engine.ts`
- **问题**: trajectories 表 + FTS5 索引无限增长，DB 膨胀后 WASM 堆无法归还
- **方案**:
  1. Database 新增 `cleanupOldTrajectories(daysToKeep = 90)` 方法：`DELETE FROM trajectories WHERE created_at < datetime('now', '-90 days')`
  2. 在 `Database.create` 后调用一次清理
  3. `getProjectTrajectories` 的 10000 限制改为 500 分页
  4. 可选：设置 `setInterval` 每 24h 自动清理一次
- **验证**: 测试清理后 DB 体积减小

### 阶段 3：缓存层建设（中影响）

#### 任务 3.1：Repository 层 LRU 缓存
- **文件**: 新建 `src/main/memory/lru-cache.ts`、修改 `src/main/memory/repositories/base-repository.ts`
- **问题**: DB 查询无缓存，每次切换页面重新查询 + 序列化 + IPC 传输
- **方案**:
  1. 创建轻量 LRU 缓存类（不引入外部依赖）：`class LRUCache<K, V> { private map: Map<K, V>; private max: number; private ttl: number }`
  2. BaseRepository 增加 `protected cache: LRUCache<string, any>`
  3. `getById` 命中缓存直接返回，`run`（写入）时 invalidate 对应 key
  4. max 100 entries, ttl 60s
- **验证**: 测试连续 getById 只查一次 DB

#### 任务 3.2：chapterList 字段拆分
- **文件**: `src/main/memory/repositories/chapter-repository.ts`、`src/preload/index.ts`、相关 renderer 代码
- **问题**: chapterList 返回所有章节含 content（TipTap JSON），单次可返回几十 MB
- **方案**:
  1. `listByNovel` 的 SQL 只 SELECT `id, title, sort_order, word_count, status, updated_at`（不含 content）
  2. 新增 `listByNovelWithContent` 方法供需要 content 的场景使用
  3. renderer 端列表展示用精简版，编辑器打开章节时单独 getChapter(id)
- **验证**: 测试 chapterList 返回体积减小

### 阶段 4：资源释放改进（中影响、可并行）

#### 任务 4.1：WritingGoal timer 清理 + simulateStreamingResponse 可取消
- **文件**: `src/renderer/components/editor/extensions/WritingGoal.ts`、`src/renderer/components/ai-chat/AIChatView.tsx`
- **方案**:
  1. WritingGoal 添加 `onDestroy()` 清理 `_debounceTimer`
  2. `simulateStreamingResponse` 接受 `AbortSignal` 参数，循环中检查 `signal.aborted`
  3. AIChatView 的 unmount cleanup 中调用 abort
  4. 行 251 的 setTimeout 用 ref 持有并在 unmount 清理
- **验证**: 测试 unmount 后无 setTimeout 回调执行

#### 任务 4.2：chatSlice beforeunload flush + pendingMessages 修复
- **文件**: `src/renderer/store/chatSlice.ts`
- **问题**: saveTimer 无 unload flush 可能丢消息；pendingMessages 流式期间过时
- **方案**:
  1. store 初始化时注册 `window.addEventListener('beforeunload', () => flushSave())`
  2. `setStreaming(false)` 时先更新 pendingMessages 引用再 flushSave
  3. `appendToMessage` 流式分支也更新 pendingMessages（或改为 flush 时从 store 获取最新）
- **验证**: 测试流式结束后保存的是最新消息

#### 任务 4.3：LLM SSE 解析优化
- **文件**: `src/main/engine/llm-provider.ts`
- **问题**: `buffer += chunk; buffer.split('\n')` 每次都创建新字符串和数组，O(n²)
- **方案**:
  1. 改为维护 "未完成行" 边界指针，用 `indexOf('\n')` 增量解析
  2. 或使用 `TextDecoderStream` + `TransformStream` 管道流式解析 SSE
  3. done 分支添加显式 `reader.releaseLock()`
- **验证**: 测试长流式响应（1000+ chunks）的内存增长

### 阶段 5：内存监控体系（低影响、可并行）

#### 任务 5.1：内存监控 IPC + 阈值预警
- **文件**: 新建 `src/main/ipc/monitor.ipc.ts`、修改 `src/preload/index.ts`、修改 `src/main/ipc/index.ts`
- **方案**:
  1. 新增 `monitor:memory-usage` handler，返回 `process.memoryUsage()` + DB 大小
  2. main 进程 `setInterval`（60s）采集内存，超阈值（如 heapUsed > 500MB）时 log.warn
  3. preload 暴露 `getMemoryUsage()` 方法
  4. renderer 端可选：Settings 页面显示内存使用情况
- **验证**: 测试监控数据正确返回

#### 任务 5.2：IPC payload 大小日志
- **文件**: `src/main/ipc/index.ts`
- **方案**: 在 `wrap()` 中添加 `JSON.stringify(args).length` 计算，超 1MB 时 `log.warn` 告警
- **验证**: 测试大 payload 触发告警

### 阶段 6：代码注释与文档

#### 任务 6.1：内存密集型操作注释
- **文件**: `database.ts`、`llm-provider.ts`、`operation-log.ts`、`chatSlice.ts`、`AIChatView.tsx`
- **方案**: 在每个内存密集型操作处添加注释，标注：

  - 潜在内存风险点
  - 优化思路
  - 资源生命周期说明

### 阶段 7：压力测试与验证

#### 任务 7.1：内存压力测试
- **文件**: 新建 `tests/integration/memory-stress.test.ts`
- **方案**:
  1. 模拟高频 DB 写入（1000 次 chapterUpdate）
  2. 模拟大流式响应（5000 chunks SSE）
  3. 模拟并发 LLM 请求
  4. 验证内存增长在可接受范围

## 假设与决策

| 决策 | 说明 |
|------|------|
| LRU 缓存自行实现 | 不引入 `lru-cache` npm 包，保持依赖精简（当前仅 5 个运行时依赖） |
| trajectories 保留 90 天 | 平衡学习分析数据需求与 DB 体积 |
| chapterList 拆分 content | 列表展示不需要 content，编辑器打开时单独获取 |
| abort 使用 requestId | 而非全局单个 controller，支持未来多请求并发 |
| 监控用 setInterval 60s | 平衡监控实时性与 CPU 开销 |
| soft_heap_limit 50MB | sql.js 默认无限制，50MB 适合中型小说项目 |
