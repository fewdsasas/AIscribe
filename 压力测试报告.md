## 五、性能瓶颈识别

### 5.1 已识别瓶颈

| 瓶颈 | 严重度 | 位置 | 影响 |
|------|--------|------|------|
| OperationLog 同步 flush 阻塞 | 中 | operation-log.ts append() | pending 触顶时同步 I/O 阻塞主线程 |
| LRU 写后全量 clearCache | 低 | base-repository.ts run() | 高写场景缓存命中率趋零 |
| save() 全量 export | 低 | database.ts save() | 大 DB 时 save 内存峰值（已优化 3x→2x） |
| activeControllers Map 无上限 | 低 | llm-provider.ts | 不传 requestId 不注册，传了需手动 cancel |

## 六、优化建议

### 6.1 已实施的优化（本轮验证有效）

| 优化 | 压测验证结果 |
|------|------------|
| save() 去二次复制 | 1000 次写入 save 无额外内存峰值 |
| OperationLog MAX_PENDING=1000 | 5000 条记录 pending 严格 ≤ 1000 |
| LRU 缓存 max=100 | 5000 次读写 size 严格 ≤ 100 |
| SSE 增量解析 | 5000 chunks 线性增长，无 O(n²) |
| chapterList 摘要模式 | 1000 章批量插入仅 +6.8MB |
| abort requestId 管理 | 10 路取消 5 路，activeControllers 清零 |
| Trajectory TTL 清理 | 3ms 完成，无内存影响 |

### 6.2 后续优化建议

| 优先级 | 建议 | 预期收益 |
|--------|------|---------|
| 中 | OperationLog flush 改为异步非阻塞 | 消除 pending 触顶时的同步 I/O 阻塞 |
| 低 | LRU clearCache 改为按 key invalidate | 高写场景提升缓存命中率 |
| 低 | activeControllers 加上限保护 | 防止极端场景 Map 无限增长 |
| 低 | 定期 `global.gc()` 调度 | 长时间运行后回收碎片 |

## 八、结论

系统在高并发场景下表现**稳定可靠**：

- **响应时间**: 15 个场景全部 < 200ms（含目标 < 200ms），最高 1128ms（含 500ms 等待）
- **内存占用**: 所有场景峰值增长 < 30MB，无内存泄漏迹象
- **错误率**: 0%，全部 606 个测试通过
- **吞吐量**: DB 写入 8.4K ops/s，LRU 142K ops/s，SSE 解析 357K chunks/s
- **防抖合并**: 1000 次写入 → 1 次 save，合并率 1000:1
- **资源清理**: 流取消后 activeControllers 清零，unmount 后 timer/ref 清理
